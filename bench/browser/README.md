# Browser bench

Real-browser full-load timings (Node V8 ≠ browser). Times the whole consumer
flow — `parseTextured` decoding every IGC texture — over a **real `.gr2`
corpus** you point it at, cold + warm, on two axes:

- **main** — decode on the main thread (what blocks the render loop)
- **worker** — decode in a module Worker (like roBrowser), main thread free

Same build, same JIT on both axes: the Worker isn't a faster kernel, it keeps
the main thread responsive. The **max main-thread stall** metric (frame-gap
monitor) is the render hitch a user would feel — watch it collapse on the
worker axis. `ready()` instantiation is timed separately (≈ 0 for pure JS,
real once the WASM build lands).

```sh
cd ..                                   # granny-ro-js/
npm run build                           # produce dist/granny-ro.esm.js
GR2_FOLDER=/path/to/extracted/gr2 npm run bench:browser   # stage build + corpus, print serve cmd
npx http-server bench/browser -p 8888   # then open http://localhost:8888 in Chrome/Firefox
```

`GR2_FOLDER` must hold **loose extracted `.gr2`** (assets still inside a GRF
need extraction first). `granny-ro.esm.js`, `fixtures/`, `fixtures.json`, and
`meta.json` are regenerated by `npm run bench:browser` and gitignored.

Add `?warm=N` to the URL to change warm iterations (default 20).

## Comparing two versions (Δ%)

Each run's **⬇ Download JSON** carries a `version` stamp (`meta.json`: the
staged bundle's git sha + optional `--label`), so a saved batch says which code
it measured. Drop **1 or 2** saved batch JSONs onto the page's drop zone:

- **two files** → diff them (file 1 = baseline → file 2 = current), Δ% per axis + per fixture;
- **one file** → diff it (baseline) against this page's fresh run.

Green = faster, red = slower. To compare a release against your branch — **no
`git checkout` needed**, so you never lose your branch or this tooling: `--ref`
builds that ref's bundle in a throwaway git worktree.

```sh
# baseline bundle (the release) — built in a throwaway worktree, staged + labelled
GR2_FOLDER=/path/to/gr2 npm run bench:browser -- --ref=v1.2.0
npx http-server bench/browser -p 8888
#   open http://localhost:8888?warm=15 → ⬇ Download JSON   (the baseline batch)

# your branch's bundle (current working tree — all your changes)
npm run build
GR2_FOLDER=/path/to/gr2 npm run bench:browser -- --label=now
npx http-server bench/browser -p 8888
#   open http://localhost:8888?warm=15 → drop the v1.2.0 JSON → Δ% table
```

For a scripted node-side equivalent (10-15 runs/side, averaged, Δ% with σ), use
`npm run perf:regress -- --baseline=v1.2.0 --runs=12` from the repo root.

## Axes

The `AXES` array in `bench.js` pairs a build URL + thread mode with an **op**
(from the `OPS` map). Two comparisons:

- **JS-vs-WASM** (`op: parseTextured`) — `js-esm` / `wasm-esm`, each × main / worker.
- **single-pass** (WASM build) — `wasm-esm · load3x · {main,worker}` vs
  `wasm-esm · load1x · {main,worker}` :
  - **`load3x`** reproduces roBrowser's `GR2Loader.load()` today — `parseTextured`
    + `parseAnimated` + `extractModels(loadGR2(parseGR2File(u8)))`, three `loadGR2`.
  - **`load1x`** is the single-pass `parseAll(u8)` — one `loadGR2`, every extractor.

The verdict card (and exported payload `summary.load1xVs3x`) reports the
`load3x / load1x` ratio per thread mode — the 3×→1× parse win.
